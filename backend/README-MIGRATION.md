# Работа с миграциями Alembic

## TL;DR

```bash
alembic upgrade head       # применить все миграции
alembic check               # убедиться, что модели и БД синхронны
alembic revision --autogenerate -m "описание"   # создать новую после изменения моделей
```

## Важно: как выглядит история сейчас

`001_initial_schema` — единая консолидированная миграция, создающая всю схему
БД целиком. Она появилась при переработке проекта, потому что старая история
миграций была нерабочей (несколько независимых веток, миграция, пытавшаяся
дропнуть несуществующую таблицу) — `alembic upgrade head` падал на чистой БД.
Сейчас это исправлено, и `001_initial_schema → 005_add_roles` — единственная,
линейная и проверенная цепочка.

**Больше не удаляйте файлы миграций вручную, если в этом нет крайней
необходимости.** Если модель изменилась — создавайте новую миграцию поверх
существующих (`alembic revision --autogenerate`), а не пересоздавайте историю
с нуля. Пересоздание истории оправдано только если история снова окажется
сломана (например, при слиянии веток с расходящимися миграциями) — тогда
стоит явно предупредить команду и синхронизировать все окружения (dev/staging/prod).

## Частые проблемы

### "Multiple head revisions are present"

Значит, в `alembic/versions/` есть две миграции с одинаковым (обычно `None`)
`down_revision`, то есть два независимых корня истории. Нужно явно прописать
`down_revision` одной из них так, чтобы она указывала на другую — сделав
историю линейной. Проверить количество "голов":

```bash
alembic heads
```

### "Target database is not up to date"

В БД есть непримененные миграции.

```bash
alembic current      # текущая версия в БД
alembic history       # полная история
alembic upgrade head  # применить все pending
```

Если после `alembic upgrade head` ошибка не уходит — вероятно, реальная схема
БД разошлась с тем, что думает Alembic. Проверить:

```bash
alembic current
```

Если она показывает `(head)`, но всё равно ругается — можно принудительно
пометить текущее состояние как актуальное (только если вы **точно** знаете,
что схема БД уже соответствует этой ревизии):

```bash
alembic stamp head
```

### Как проверить, что модели и БД не разошлись

```bash
alembic check
```

Ожидаемый вывод: `No new upgrade operations detected.` Если что-то выводится —
значит, кто-то поменял модель, но не сгенерировал миграцию (или наоборот).

## Локальная разработка через Docker

```bash
docker exec -it velo_backend bash
alembic revision --autogenerate -m "описание изменения"
alembic upgrade head
```

Обратите внимание: сам контейнер (см. `entrypoint.sh`) уже применяет
`alembic upgrade head` при каждом старте — вручную это нужно делать только
при создании новой миграции в работающем контейнере.
