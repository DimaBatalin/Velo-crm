# Контекст проекта для AI-ассистента

Этот файл — брифинг для нового диалога с AI-ассистентом (Claude Code или
аналог), чтобы продолжить работу над Velo-CRM без повторного объяснения
контекста пользователем. Вставьте/приложите этот файл в начале нового
диалога. Подробности по коду — в [`Documentation.md`](Documentation.md),
здесь — только оперативный контекст и грабли, на которые уже наступали.

## Что это за проект

Velo-CRM — CRM для велопроката/велосервиса. FastAPI (async, Postgres,
Alembic) + Vue 3 (без роутера, Vite). Полная модель данных, API и бизнес-правила
описаны в `Documentation.md` — прочитай его, если нужно менять backend-логику
или структуру данных.

## Как поднять окружение

```bash
docker compose up --build          # backend :8000, frontend(nginx) :3000, postgres
```

Учётка админа по умолчанию (из `.env`): `velo@velo.local` / `changeme123`.

**Для активной фронтенд-разработки** удобнее не трогать docker-frontend, а
поднять Vite dev-сервер поверх backend'а из Docker:
```bash
cd frontend && npm install && npm run dev    # :5173, HMR
```
CORS для `localhost:5173` уже захардкожен в `backend/app/main.py` — работает
из коробки, ничего в `.env` для этого добавлять не нужно (`ALLOWED_ORIGINS`
в `.env` вообще не используется, см. `Documentation.md` §9).

## ⚠️ Главная ловушка: Docker без bind-mount

`docker-compose.yml` собирает `backend`/`frontend` из `Dockerfile` **без
монтирования исходников** — код копируется внутрь образа один раз при
сборке. После правок backend-кода:

```bash
docker compose restart backend       # НЕ подхватит изменения — тот же образ!
docker compose up -d --build backend # правильно — пересобирает образ
```

Это уже один раз стоило часа отладки (правка Pydantic-схемы "не применялась",
хотя файл был точно изменён) — не наступай на те же грабли.

## Известные особенности кода (не баги, просто устройство так)

- RBAC проверяется **только на backend'е** — фронтенд показывает все формы
  всем ролям, при `403` просто всплывает тост с ошибкой.
- Статус велосипеда (`ready`/`rented`/`repair`/`stolen`) синхронизирует
  **фронтенд** явными вызовами `PUT /bikes/{id}`, backend этого не делает сам.
- `total_cost` ремонта не хранится в БД — считается на лету (сумма услуг +
  списанных запчастей по цене на момент списания).
- Нет `DELETE /tags/{id}` и `DELETE /users` — эти сущности можно только
  создавать/читать (пользователей — деактивировать вручную в БД).
- JWT живёт 8 часов (`ACCESS_TOKEN_EXPIRE_MINUTES=480`) — если токен в
  `localStorage` протух между сессиями тестирования, первый запрос вернёт
  401, просто перелогинься.

## Инструменты тестирования UI в этой среде

При работе через Claude Code с `preview_*` MCP-инструментами (не claude-in-chrome):
- **`preview_screenshot` может показывать устаревший/неверный кадр** сразу
  после `preview_resize` или сразу после клика, открывшего модалку —
  наблюдалась "схлопнутая в узкую колонку" картинка, хотя DOM
  (`getBoundingClientRect` через `preview_eval`) показывал корректную
  полную ширину. **Доверяй `preview_eval`/DOM-измерениям больше, чем
  скриншоту**, если они расходятся; при необходимости — `reload()` страницы
  обычно "чинит" скриншот.
- **Открывать модалку и сразу в том же вызове её заполнять — рискованно**:
  между `.click()` на кнопку открытия и рендером Vue должен пройти хотя бы
  один tick. Надёжнее: (1) отдельным вызовом `preview_eval` кликнуть кнопку
  открытия, (2) отдельным вызовом проверить, что модалка/инпуты
  действительно в DOM и пустые, (3) только потом заполнять и сабмитить.
  Однажды нарушение этого порядка привело к тому, что в БД создалась
  запчасть с мусорными значениями полей ("22"/"2"/...) — сейчас это
  исправлено, но паттерн стоит помнить.
- Логин через форму: `preview_fill` на `input[type="email"]` и
  `input[type="password"]`, затем клик `button[type="submit"]`.
